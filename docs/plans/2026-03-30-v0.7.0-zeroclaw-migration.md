# Project Black-Ice: v0.7.0 "Edge-Compute" Migration Plan
**Date:** March 30, 2026
**Target:** Node A (Rules Authority) & Node B (Orchestrator)
**Status:** SLATED (Post-v0.6.0 completion)

## 1. Objective
Decommission the high-overhead Docker/PostgreSQL stack on **Node A (192.168.0.50)** and replace it with a high-performance, Rust-native **ZeroClaw** implementation. Establish a sub-10ms **ClawLink** bridge to **Node B** and implement a **Unified Oracle** state manager on the main rig to eliminate narrative drift and hardware latency.

---

## 2. Technical Architecture: The "Triple-SQLite" Stack
To align with the **Crush CLI's** native SQLite usage and the **Local-Only** mandate, the project will consolidate onto a unified SQLite data plane.

| Component | Database | Role | Storage Location |
| :--- | :--- | :--- | :--- |
| **Node A (ZeroClaw)** | `rules.db` | 1,437 Vector Chunks + Mechanics | `~/zeroclaw/rules.db` (Node A) |
| **Node B (Orchestrator)** | `world.db` | Relational Knowledge Graph (RKG) | `D:\asp-gm-agent\world.db` (Node B) |
| **Crush CLI** | `crush.db` | Session State & Chat History | `.crush/crush.db` (Node B) |

---

## 3. Implementation Phases

### **Phase 1: ZeroClaw Deployment (Node A)**
- **Binary:** Build/Deploy the Rust-native ZeroClaw binary to Node A.
- **SQLite-Vec:** Initialize the `vec0` virtual table with FTS5 integration for Hybrid Search (Keyword + Vector).
- **Migration:**
    - Export `nitro_db.pdf_chunks` from Postgres on Node A to CSV.
    - Import CSV into ZeroClaw `rules.db` using the `zeroclaw-import` utility.
    - Verify 100% chunk parity (1,437 vectors).

### **Phase 2: ClawLink Protocol (The Bridge)**
- **Transport:** Replace the SSH-over-Stdio transport with **ClawLink** (Persistent Socket-over-SSH).
- **Authentication:** Configure Ed25519-based access tokens for secure, long-lived Node B -> Node A communication.
- **Latency Goal:** <10ms round-trip for Rules Oracle queries.

### **Phase 3: Unified Oracle (RKG on Node B)**
- **Schema:** Implement a **Subject-Predicate-Object (Triplet)** schema in `world.db` on Node B.
- **Grounding Logic:** Update the `StoryEngine` to query the Knowledge Graph before generating narrative prose, preventing AI hallucinations.
- **Crush Integration:** Use SQLite's `ATTACH DATABASE` to allow the Oracle to join the Knowledge Graph with the Crush session history.

## 5. Success Criteria
1. **Performance:** Sub-10ms latency for mechanical math resolution.
2. **Reliability:** 100% parity with existing `nitro-db` and `nitro-logic` tools.
3. **Efficiency:** Reclaim ~600MB of RAM on Node A by decommissioning Docker/Postgres.
4. **Drift Prevention:** Narrative Engine (Mistral-Nemo) consistently verifies world facts against the RKG.
