✅ ANALYSIS – MoshiRAG (https://github.com/kyutai-labs/moshi-rag + arXiv:2604.12928)
Potential as Our Voice Engine in Beta v3

Short Answer
Yes — MoshiRAG has strong potential as a full-duplex voice engine component in our stack. It is not a drop-in replacement for the entire voice pipeline, but it is an excellent candidate to replace or augment the current Hermes Whisper STT + VoxCPM2 TTS layer with native, real-time, factual speech-to-speech capabilities.

1. What MoshiRAG Actually Is

  • Base Model: Built on Moshi (Kyutai Labs’ full-duplex speech language model).
  • Core Innovation: Adds asynchronous RAG (Retrieval-Augmented Generation) without breaking real-time interactivity.
    • The model predicts when external knowledge is needed.
    • It triggers retrieval in the background during natural pauses.
    • Retrieved references are injected back into the conversation stream without interrupting speech output.
  • Architecture:
    • Front-end: Moshi handles full-duplex speech (listen + speak simultaneously).
    • Back-end: Separate retrieval service (LLM or search-based) that runs in parallel.
    • Reference Encoder: Encodes retrieved text and injects it into Moshi’s conditioning path.
  • Key Features:
    • Full-duplex (no push-to-talk).
    • Low-latency factuality grounding.
    • Streaming ASR (Gradium or local).
    • Plug-and-play retrieval back-ends.
    • Web UI for testing.
  • Performance: Designed for real-time conversational AI. Retrieval latency target < 3 seconds to maintain interactivity.

Paper Highlights (arXiv:2604.12928)
- Focuses on solving factuality issues in full-duplex speech models.
- Achieves factuality comparable to non-duplex models while preserving interactivity.
- Strong on out-of-domain tasks (e.g., math reasoning) via RAG.

2. POTENTIAL USE CASES AS OUR VOICE ENGINE

Primary Use Case
Replace the current Hermes Whisper STT → VibeVoice → VoxCPM2 TTS pipeline with MoshiRAG as the core full-duplex voice engine, then layer Hermes orchestration on top for higher-level agentic control.

Why This Is Strong
- Native full-duplex speech (listen and speak at the same time) — better than our current turn-based pipeline.
- Built-in asynchronous RAG → factual grounding without latency penalty (perfect for live coding partner or general conversations).
- Real-time, streaming, and controllable — aligns with our Omi hardware and PC mic input goals.
- Can be run locally (PyTorch or Rust backends) → fully sovereign.

Concrete Integration Options in Beta v3

Option 1 (Recommended – Modular Appendage)
- Run MoshiRAG as a pluggable voice engine module in the monorepo (crates/modules/moshi-rag/).
- Input: Omi wearable / PC headset mic (via Clawlink bridge).
- Processing: MoshiRAG handles full-duplex STT + RAG + TTS.
- Output: Audio streamed back to Omi/PC + logged to MemPalace.
- Hermes (patched fork) acts as the higher-level orchestrator (routes tasks, uses GEPA reflection, etc.).
- Pretext HUD shows live voice status and RAG references.

Option 2 (Hybrid)
- Keep Hermes Whisper + VoxCPM2 for simple voice tasks.
- Switch to MoshiRAG for knowledge-intensive or long conversations (dynamic routing via model router).

Benefits for Our Stack
- Improves factuality in voice interactions (critical for the live coding partner).
- Reduces latency compared to separate STT + TTS steps.
- Fits our modular beta v3 philosophy (pluggable module, easy to enable/disable).
- Complements VibeVoice (use VibeVoice for post-processing if needed) and VoxCPM2 (use VoxCPM2 for voice design/cloning if MoshiRAG TTS is not sufficient).

Risks / Limitations
- MoshiRAG requires a capable GPU (24 GB+ recommended for full-duplex; smaller setups possible with Rust backend).
- It is newer and less battle-tested than our current pipeline.
- Integration requires some work (MCP bridge + audio streaming).
