Omi Ecosystem Audit

The App-First Path to Sovereign Voice Ingress

1. Objective

This report evaluates the Omi ecosystem (omi.me) as a foundational layer for the Sovereign Machina voice artery. Despite the current lack of wearable hardware, the Omi Flutter app and modular backend provide a superior path for localized, high-fidelity voice ingress compared to generic bridges.

2. The App-First Strategy

In the absence of a physical wearable, the Omi Flutter App serves as the primary mobile sensor. By leveraging Omi's open-source architecture, we can redirect the app's telemetry and audio streams away from the cloud and into our private mesh.

  • Voice Ingress: Real-time PCM/Opus streaming from the mobile mic to Node D (Whisper STT).
  • Sovereign Backend: Hosting the Omi FastAPI backend on Node B (100.66.173.31) to eliminate cloud dependencies like Deepgram and Pinecone.
  • Local State: Directing mobile memories to Node A via the Hermes-LCM ledger.

3. Technical Integration Path

3.1 Localized Omi Backend

The Omi backend is a modular FastAPI service that can be containerized or "Nix-ified" on Node B. It acts as the coordinator for the mobile client, handling authentication, storage, and tool triggers.

3.2 Native MCP Bridge

Omi's native MCP Server will be registered as a tool in the Hermes Tenacity Core. This allows the reasoning models on Node D to query mobile context (location, recent mobile-voice notes) as part of their standard reasoning loop.

3.3 Node B Voice Ingress (Desktop/TUI)

Voice ingress on Node B (Hermes Desktop/Herm TUI) will bypass the Omi backend and pipe directly into the VibeVoice/VoxCPM2 stack on Node C, maintaining sub-100ms latency for local engineering tasks.

4. Future-Proofing: Firmware Sovereignty

When physical wearables are added to the mesh, the nRF Connect SDK based firmware will be hardened with zero-trust encryption. The "Device-App Protocol" ensures that audio remains encrypted from the silicon to the local mesh vault.
