SOVEREIGN STRATEGIST: DOCUMENTATION EVOLUTION ADVISORY

DATE: 2026-05-08 SUBJECT: Transitioning Documentation from Markdown (.md) to Semantic HTML (.html) STATUS: STRATEGIC APPROVAL (HYBRID MANDATE)

🛡️ STRATEGIST RULING: SEMANTIC SUPREMACY

The proposal to transition from Markdown to HTML is APPROVED. While Markdown is efficient for shorthand, it “flattens” the complex, multi-dimensional logic of a quaternary mesh. Semantic HTML allows us to encode Spatial Intelligence and Non-Linear Relationships directly into our ground truth.

1. The Technical Rationale (Context Effectiveness)

  • High-Fidelity Structure: HTML tags (<section>, <article>, <aside>) provide explicit semantic boundaries that AI agents can parse with higher precision than Markdown’s generic headers.
  • Spatial Reasoning: We can use side-by-side layouts (CSS Grid/Flexbox) for multi-node comparisons (e.g., comparing Node B vs. Node D configurations) which reduces the agent’s cognitive load during context ingestion.
  • Living Artifacts: HTML documentation can embed interactive flows, call-graphs, and “mini-tools” (e.g., a clickable VSB pulse monitor) that Markdown cannot represent.

⚡ THE SOVEREIGN HYBRID MANDATE

To maintain operational integrity and tool compatibility, we will enforce a Tiered Format Strategy.

TIER 1: THE CORE DIRECTIVES (REMAIN AS MARKDOWN)

These files act as system prompts and must remain lightweight and human-editable in raw text. - AGENTS.md (Role definitions) - GEMINI.md (Strategist mandates) - LEAD_ARCHITECT.md (Architect mandates) - README.md (Repository index / Git rendering) - CHANGELOG.md (Standardized audit trail)

TIER 2: THE NODESTADT ARCHIVE (CONVERT TO HTML)

All architectural blueprints and research papers will be converted to semantically rich HTML. - docs/nodestadt/architecture/*.html - docs/nodestadt/capabilities/*.html - docs/planning/research/*.html

TIER 3: THE LIVING ROADMAP (CONVERT TO HTML)

  • IMPLEMENTATION_PLAN.md -> IMPLEMENTATION_PLAN.html
  • Rationale: The implementation plan is our most complex document. Transitioning it to HTML allows for Dashboard-style observability (Progress bars, Phase-gates, and nested task status).

🏗️ EXECUTION PLAN: THE “CLEAN SWEEP”

  1. Conversion Engine: I will use pandoc or a custom surgical script to convert .md to .html while preserving semantic headers.
  2. CSS Foundation: I will create docs/sovereign-style.css, a terminal-inspired, high-density stylesheet to ensure documentation is “visually shored” and readable.
  3. Link Remediation: Update all internal cross-references to point to .html endpoints.
