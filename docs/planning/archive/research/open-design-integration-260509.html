Phase 6 Research Report

Open Design Integration & Pretext HUD Materialization

1. Objective

This report evaluates the Open Design engine (nexu-io/open-design) as the primary materialization layer for Phase 6. The goal is to determine the feasibility of forking this design-to-code engine and injecting the Pretext kinetic typography protocol into its core rendering loop without breaking architectural integrity.

2. Architectural Audit

2.1 Delegated Reasoning Model

Open Design operates on a Delegated Reasoning Model, offloading design-to-code logic to external coding agents (Claude Code, Codex, etc.). It manages state via a local SQLite database and streams artifacts using Server-Sent Events (SSE). This architecture is perfectly aligned with our Node D (Logic) and Node B (Control) split.

2.2 Decoupled UI

The frontend is a modern Next.js 16 application. Visual aesthetics are governed by deterministic "Directions" defined in prompts/directions.ts, utilizing OKLch color spaces and font stacks. This modularity allows for the native injection of our Operational Lo-Fi Brutalism aesthetic.

3. Pretext HUD Materialization

The primary concern—modifying the UI with Pretext without breaking the engine—is addressed by the system's Skills Protocol.

  • Injection Point: Pretext can be injected as a high-fidelity rendering skill. By wrapping the Pretext library into a custom SKILL.md, we can force the reasoning engine to utilize kinetic typographic layouts for all generated artifacts.
  • Rendering Loop: Since Open Design utilizes a sandboxed srcdoc iframe for real-time previews, Pretext's JavaScript-based typography engine can execute in total isolation from the Next.js frontend logic.

4. Distributed Mesh Deployment (Quaternary)

5. Feasibility Verdict

Verdict: HIGHLY FEASIBLE.

Open Design's "BYOS" (Bring Your Own Skill) philosophy makes it the ideal chassis for a Sovereign Design Artery. We can heavily modify the visual output using Pretext while preserving 100% of the underlying design-to-code reasoning logic.
