✅ HONEST & UNBIASED ASSESSMENT – Should We Fork Omi Like We Did Hermes?

Short Answer
No — it does not make sense to fork the entire Omi repo at this stage.

Forking Omi would be overkill and add unnecessary maintenance burden, especially as we are deliberately simplifying the architecture for the beta v3 rebuild.
The Omi team is indeed very active, which is actually a reason not to fork right now.

PROS / CONS COMPARISON

Aspect Forking Omi (Like Hermes) Keeping Official Omi + Extending Our Layer Winner
Control Full control over app and future features Good control via our Machina Terminal (Flutter) Tie (but fork is heavier)
Maintenance High — must rebase frequently on active upstream Low — we only maintain our thin extension layer Official + Extension
Development Velocity Slower (merge conflicts, testing full fork) Faster (focus only on our custom features) Official + Extension
Risk High — could break Omi hardware compatibility Low — Omi SDK/MCP bridge is stable Official + Extension
Alignment with Beta v3 Goal Contradicts “lean into Hermes routes, simplify” Supports clean, modular, Hermes-first rebuild Official + Extension
Long-term Scalability Harder to keep in sync with active Omi team Easier — we benefit from Omi updates automatically Official + Extension

Key Insight
Omi is input hardware + client app.
Hermes is the core agent runtime.
We forked Hermes because it is the brain of the system — we needed deep control and sovereign patches.
Omi is the peripheral input layer. We do not need to own the entire stack; we only need a clean bridge to it.

RECOMMENDED APPROACH (Most Practical for Beta v3)

Do NOT fork the full Omi repo.

Instead:

  1. Use the official Omi app + Omi SDK / MCP bridge as-is (already mined and stable).
  2. Extend our own Machina Terminal (the Flutter app already in the monorepo at terminal-app/) as the custom sovereign layer on top of Omi.
  3. Keep the Clawlink bridge (already in monorepo) as the secure RPC layer between Omi and Node D.
  4. Amplify voice pipeline (Hermes Whisper → VibeVoice → VoxCPM2) on Node D, as previously designed.

This gives us: - Full control over the Machina-specific experience (custom UI, commands, voice feedback).
- Automatic benefit from Omi team’s active development (new hardware features, bug fixes, app improvements).
- Minimal maintenance — we only maintain our thin Flutter extension and the Clawlink bridge.
- Clean alignment with the beta v3 goal of simplicity and modularity.

How This Fits the Beta v3 Rebuild
- The Flutter Machina Terminal becomes a pluggable appendage module.
- No changes to the Hermes submodule or core brain.
- Everything stays in the monorepo and follows the same submodule/pinning rules.
- Cyberpunk Red and other future plugins remain isolated.

This is the practical, long-term maintainable choice.
