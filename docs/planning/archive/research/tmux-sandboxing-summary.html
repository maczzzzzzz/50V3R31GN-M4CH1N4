Managed Tmux & Zeroboot Sandboxing: Research Summary

Date: 2026-05-01 Source: Grok Research Series (v2.1.0 Archive) Objective: Establish a lightweight, zero-overhead isolation layer for stacking multiple Hermes sessions without context bleed.

1. The Tmux Namespace Boundary

  • Thesis: Use tmux as the simplest separation layer.
  • Standard: One tmux session per project/model/role.
  • Isolation: Separate CWD, environment variables, and Hermes conversation history per session.
  • Persistence: Uses tmux-resurrect or custom session savers for reboot recovery.

2. Managed Session Fabric (Node D)

  • Sidecar: sidecars/tmux-session-manager (Rust).
  • Audit: Integrated with Psy-core for hash-chained audit trails and HMAC-signed data.
  • Self-Healing: GEPA v2 monitors and restarts crashed sessions.
  • Observability: Pretext HUD panel for real-time resource tracking.

3. Zeroboot Hardware Isolation

  • Tech: KVM VM sandboxes with 0.8ms startup and 265KB footprint.
  • Performance: 190x faster than E2B; supports 1000 concurrent forks in ~815ms.
  • Security: Hardware-enforced memory isolation (Firecracker CoW snapshots).
  • Connectivity: Serial I/O only (no network stack inside sandbox) for maximum security.

4. Clawlink: Networked Sandboxing

  • Concept: Android Machina Terminal (Omi) as a universal trigger.
  • Bridge: Clawlink sidecar acts as a secure RPC bridge over Tailscale + SPIFFE.
  • Workflow: Voice command from Omi -> Clawlink RPC -> Zeroboot Sandbox on Node B/C/D.

Integration Standards

  • Every tmux session is a first-class citizen in MemPalace.
  • All activity logged as immutable DevEpisode.
  • Nix-native deployment via flake.nix services.
