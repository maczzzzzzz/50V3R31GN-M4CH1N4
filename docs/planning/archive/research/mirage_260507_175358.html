✅ DEEP ANALYSIS – Mirage (https://github.com/strukto-ai/mirage)

Repo Status (Verified 2026-05-07)
- Purpose: A unified virtual filesystem for AI agents. It mounts diverse backends (RAM, disk, S3, Google Drive, Slack, Gmail, Redis, etc.) as a single tree under /. Agents interact with everything using familiar Unix commands (bash, grep, cp, ls, etc.) without learning separate SDKs.
- License: Apache-2.0 (fully compatible with sovereign use).
- Activity: Extremely new — initial public release ~18 hours ago (v0.0.1-alpha.1), 4 commits total, 1 contributor (zechengz). 398 stars, 20 forks.
- Tech Stack: TypeScript (core) + Python SDK. Supports Redis for distributed caching and snapshotting.
- Key Architecture:
- Workspace object that manages mounts.
- Pluggable Resources (RAMResource, S3Resource, SlackResource, etc.).
- Bash-compatible execution engine (ws.execute(command)).
- Two-layer cache (index + file) with RAM or Redis backends.
- Snapshots for portable, versioned workspaces.

MULTI-PERSPECTIVE ANALYSIS FOR SOVEREIGN MACHINA (Beta v3 Context)

Strengths & Potential Use Cases
- Unified Filesystem Abstraction: Agents see one coherent tree instead of separate APIs. This is very powerful for our Hermes-centric stack — a single ls /s3/logs or grep error /slack/channel works across services.
- Bash Compatibility: Agents trained on Unix tools can operate Mirage immediately. This fits our “lean into Hermes routes” direction and reduces custom glue code.
- Portability & Snapshots: Workspaces can be snapshotted and moved between nodes. This complements Zeroboot isolation and our multi-node mesh.
- Redis Caching: Can be used for shared state across Nodes B/C/D (distributed index and file cache).
- MCP / Agent Framework Integration: Has adapters for OpenAI Agents SDK, Vercel AI, LangChain, etc. We can expose Mirage as an MCP tool or resource for Hermes.

Risks & Limitations
- Very Early Stage: Only 18 hours old as a public repo. Not battle-tested in production multi-node setups.
- Python/TS Heavy: The core is TypeScript with Python SDK. We would need to run it inside Zeroboot or wrap it as a thin Rust module to keep our NixOS purity.
- No Native NixOS / Multi-Node Focus: Designed for single-machine or cloud agent environments. We would have to add our own Nix flake packaging and cluster-aware mounting.
- Security Surface: Mounting external services (Slack, Gmail, S3) increases attack surface. We must enforce strict Psy-core audit and SPIFFE identity checks.

Overall Verdict
Mirage is useful as a pluggable module for the beta v3 rebuild, but not as a core component.
It gives us a clean, agent-friendly way to unify data sources across the cluster without writing custom SDK wrappers for every service.
It fits our modular philosophy: we can enable/disable it per node, run it in Zeroboot sandboxes, and expose it via MCP to Hermes.

Recommended Integration (Beta v3)

  • Module: crates/modules/mirage-integration/ (tracked directory in monorepo).

  • Nix Config (pluggable):

    services.mirage = {
  enable = false;                     # default off for core minimalism
  zeroboot_sandbox = true;
  redis_cache = true;                 # shared across nodes
  mounts = {
    "/s3" = "s3-resource";
    "/slack" = "slack-resource";
  };
};

  • Hermes MCP Tool Wrapper:

    #[mcp_tool]
async fn mirage_execute(
    command: String,
    context: MemPalaceContext,
) -> Result<MirageResult> {
    let result = mirage::execute(command).await?;
    let episode = DevEpisode::from_mirage_command(&result);
    context.graph.store_episode(episode).await?;
    psy_core::audit_mirage(&result).await?;
    Ok(result)
}

Use Cases in Our Stack
- Unified access to logs, Slack, Gmail, S3 from any agent session.
- Portable workspaces that can be snapshotted and moved between nodes.
- Complements MemPalace by providing a filesystem view of external data sources.

Final Recommendation
Add Mirage as an optional pluggable module in beta v3.
Do not make it core — keep the beta minimal and clean. Enable it only when we need unified multi-service filesystem access.
