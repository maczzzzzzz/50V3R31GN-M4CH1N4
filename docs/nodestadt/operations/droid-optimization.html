Droid CLI Runtime Optimization

NODESTADT Operations | Phase 3 | Last Updated: 2026-05-10

Problem Statement

Stale droid processes were causing significant performance issues on Node B:

  • Stale process (PID 1554441) running since May 09 consuming 29% CPU
  • Multiple droid instances competing for resources
  • System resources otherwise healthy (19GB RAM available, 5% disk used)
  • Temporary file accumulation (2981+ files in /tmp/droid-*)
Impact: Degraded CLI responsiveness, increased latency, and wasted computational resources.

Solution Architecture

Implemented a two-layer monitoring and cleanup system:

1. Droid Process Watchdog (droid-watchdog.sh)

Monitors droid processes and flags/kills stale instances based on configurable thresholds:

  • Max CPU: 50% (configurable)
  • Max Age: 30 minutes default (aggressive: 15min)
  • Max Memory: 2048 MB
# Usage
./scripts/droid-watchdog.sh --dry-run                    # Check without killing
./scripts/droid-watchdog.sh --max-age 15 --kill-stale    # Kill processes >15min old

2. Droid Runtime Cleanup (droid-cleanup.sh)

Comprehensive maintenance script that runs multiple checks:

  1. Stale process detection (15min threshold)
  2. Zombie process identification
  3. Droid version verification
  4. Temporary file cleanup (24h+ old)
  5. System resource health check
# Usage
./scripts/droid-cleanup.sh

0 * * * * /home/nixos/50V3R31GN-M4CH1N4/scripts/droid-watchdog.sh --max-age 15 --kill-stale >> /tmp/droid-watchdog.log 2>&1

Root Cause Analysis

Identified potential causes for stale droid processes:

  • Graceful Shutdown Failures: Droid processes not terminating cleanly on session exit
  • Orphaned Child Processes: Subprocesses (PID 18935) outliving parent sessions
  • Network Timeout Handling: Long-running operations without proper timeout mechanisms
  • Temporary File Leaks: /tmp/droid-* accumulation indicating incomplete cleanup

Best Practices for Droid Usage

  • Always run droid-watchdog.sh --dry-run before killing processes
  • Use aggressive thresholds (15min) for development environments
  • Monitor /tmp/droid-* directory growth
  • Check process hierarchy with ps -eo pid,ppid,etime,%cpu,cmd | grep droid
  • Verify droid version updates: npm list -g droid
