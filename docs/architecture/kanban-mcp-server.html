Kanban MCP Server v0.1.0

MCP server wrapping the Hermes Kanban SQLite database for cross-agent task coordination. Enables Hermes and Gemini CLI to share a single task board via the Model Context Protocol.

Architecture

+-------------+     stdio      +------------------+
| Hermes CLI   |<------------->|                  |
+-------------+                |  Kanban MCP      |
                               |  Server (FastMCP) |
+-------------+     stdio      |                  |
| Gemini CLI   |<------------->|                  |
+-------------+                +--------+---------+
                                        |
                                 SQLite  |
                               +--------v---------+
                               |  kanban.db (WAL)  |
                               |  Hermes native    |
                               +------------------+

Design Decisions

DecisionRationale
Wrap existing schemaNo migration. Read/write the native Hermes kanban.db directly.
stdio transportLocal-only. No network surface. No auth needed. Tailscale provides the mesh boundary.
FastMCP (Python)Pure Python, no binary deps. Compatible with NixOS venv. Matches Hermes stack.
WAL modeEnables concurrent reads from multiple agents without blocking writes.
Claim locks with TTLPrevents duplicate work. Locks expire automatically (default 1 hour) if an agent crashes.

Schema

Reads/writes the native Hermes kanban schema. Key tables:

  • tasks -- id, title, body, status, priority, assignee, claim_lock, claim_expires, result, workspace_*
  • task_comments -- task_id, author, body, created_at
  • task_events -- task_id, event_type, actor, payload

No migrations. The MCP server creates the schema only when bootstrapping a new board.

Configuration

Hermes (~/.hermes/config.yaml)

mcp_servers:
  kanban:
    command: .../kanban-mcp-server/.venv/bin/python
    args: ["-m", "kanban_mcp.server"]
    env:
      HERMES_KANBAN_ROOT: ~/.hermes/kanban/boards
    timeout: 30
    connect_timeout: 15

Concurrency Model

  • SQLite WAL mode allows concurrent reads from both agents.
  • Write transactions are kept short (single statements).
  • claim_task uses atomic conditional UPDATE: only claims if lock is NULL or expired.
  • Claim TTL defaults to 3600 seconds (1 hour). Configurable per call.
