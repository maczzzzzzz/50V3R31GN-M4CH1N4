Quaternary Mesh

Four physical nodes connected via Tailscale (WireGuard). Each node has a dedicated role determined by its hardware constraints. The mesh is heterogeneous by design: AMD Vulkan, NVIDIA CUDA, and CPU-only backends.

Role Assignments

Node A — Synapse (State)

No model inference. Dedicated to KV-cache spillover via TurboQuant and persistent state storage via hermes-lcm. The GTX 1050 Ti is too constrained for meaningful LLM work but can buffer quantized KV tensors.

Node B — Director (Primary Workspace)

Fast responder and workspace authority. Runs Qwopus3.5-9B Q4_K_M on the AMD RX 9060 XT (16GB) via Vulkan backend (ik_llama.cpp). Benchmarked at 93.2/33.7 t/s. Node B is the primary development interface, running under WSL2 on Windows.

Node C — Oracle (Perception)

Function-calling specialist. RTX 2060 6GB VRAM running Carnice-9B-Function-Calling i1-Q4_K_M via CUDA (sm_75). Benchmarked at 205.2/49.9 t/s. External SSD (SOVEREIGN_SOUL, 469GB) mounted at /mnt/sovereign-soul.

Node D — Quaternary (Heavy Reasoning)

CPU-only inference on Meteor Lake with 48GB DDR5. Runs Qwen3.5-35B-A3B-MTP (Q4_K_M, 20GB) at ~6.1 tok/s generation via stock llama.cpp v8983. The MoE architecture means only ~3B active parameters per forward pass, making CPU inference viable despite the 35B total. RTX 5060 Ti 16GB OC upgrade planned via OCuLink (sm_120, Blackwell). Target model: Qwen3.5-35B-A3B-MTP with native MTP heads for speculative decoding.
