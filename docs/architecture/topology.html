Quaternary Mesh

Four physical nodes connected via Tailscale (WireGuard). Each node has a dedicated role determined by its hardware constraints. The mesh is heterogeneous by design: AMD Vulkan, NVIDIA CUDA, and CPU-only backends.

Role Assignments

Node A — Synapse (State)

No model inference. Dedicated to KV-cache spillover via TurboQuant and persistent state storage via hermes-lcm. The GTX 1050 Ti is too constrained for meaningful LLM work but can buffer quantized KV tensors.

Node B — Director (Primary Workspace)

Fast responder and workspace authority. Runs Qwen 3 14B on the AMD GPU via Vulkan backend (ik_llama.cpp). Also hosts Qwen3-VL-2B for vision triage. Node B is the primary development interface, running under WSL2 on Windows.

Node C — Oracle (Perception)

Perception and function-calling node. The RTX 2060 has 6GB VRAM — suited for 9B-class models at Q4 quantization. Planned to run Carnice-9B-Function-Calling for tool use and Qwen3-VL-2B for vision backup.

Node D — Quaternary (Heavy Reasoning)

CPU-only inference on Meteor Lake with 48GB DDR5. Runs Carnice-Qwen3.6-MoE-35B-A3B (Q4_K_M, 20GB) at ~6.1 tok/s generation. The MoE architecture means only ~3B active parameters per forward pass, making CPU inference viable despite the 35B total. Planned dual-model stack with a 7B interactive model on a separate port.
